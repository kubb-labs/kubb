# Using Kubb output

Kubb writes generated code under the `output.path` from `kubb.config.ts` (for example `./src/gen`),
with one folder per plugin (`models`, `clients`, `hooks`, `schemas`, `mocks`). Every generated file
starts with `Generated by Kubb. Do not edit manually`. Treat the output directory as read only: it is
overwritten on every `kubb generate`. To change generated code, change the spec or the config and
regenerate, then typecheck so any broken call sites surface.

Names are derived from the spec. Functions and hooks use the operation's `operationId` in camelCase
(`findPetsByTags`, `deletePet`), and types and schemas use PascalCase model names (`Pet`).

## Find what was generated

Before importing, learn the layout instead of guessing:

1. Read `kubb.config.ts` for the top-level `output.path` and each plugin's `output.path`.
2. List the output directory to see the real file and export names.
3. For a plugin's exact options, defaults and dependencies, read its `extension.yaml` straight from
   the install: `node_modules/@kubb/plugin-<name>/extension.yaml`. It ships with the package, so it
   matches the installed version. It carries the `npmPackage`, the `dependencies` (for example
   `plugin-client` depends on `plugin-ts`), the default `output.path`, and the full `options` schema
   with descriptions and defaults. Use it as the source of truth rather than assuming an option name.

Import from a folder's `index.ts` barrel when `output.barrel` is set in the config, otherwise import
the file directly.

## Rules

- Never edit files under `output.path`. Change the spec or config and run `kubb generate` again.
- Do not copy generated code into hand-written modules. Import it so a regenerate keeps callers in
  sync.
- Keep the runtime client configured in one place. The client functions and the framework hooks all
  share it.
- After regenerating, run a typecheck so spec changes that broke a call site show up.
