# Using Kubb output

Kubb writes generated code under the `output.path` from `kubb.config.ts` (for example `./src/gen`),
with one folder per plugin (`models`, `clients`, `hooks`, `schemas`, `mocks`). Every generated file
starts with `Generated by Kubb. Do not edit manually`. Treat the output directory as read only: it is
overwritten on every `kubb generate`. To change generated code, change the spec or the config and
regenerate, then typecheck so any broken call sites surface.

Names are derived from the spec. Functions and hooks use the operation's `operationId` in camelCase
(`findPetsByTags`, `deletePet`), and types and schemas use PascalCase model names (`Pet`).

## Find what was generated

Before importing, learn the layout instead of guessing:

1. Read `kubb.config.ts` for the top-level `output.path` and each plugin's `output.path`.
2. List the output directory to see the real file and export names.
3. For a plugin's exact options, defaults and dependencies, read its `extension.yaml` straight from
   the install: `node_modules/@kubb/plugin-<name>/extension.yaml`. It ships with the package, so it
   matches the installed version. It carries the `npmPackage`, the `dependencies` (for example
   `plugin-client` depends on `plugin-ts`), the default `output.path`, and the full `options` schema
   with descriptions and defaults. Use it as the source of truth rather than assuming an option name.

Import from a folder's `index.ts` barrel when `output.barrel` is set in the config, otherwise import
the file directly.

## What each plugin emits

### Types, `@kubb/plugin-ts`

Per operation you get types such as `FindPetsByTags`, `FindPetsByTagsQueryParams`,
`FindPetsByTagsStatus200`, `FindPetsByTagsData` and `FindPetsByTagsResponse`, plus a type per model
(`Pet`). Use them to type your own functions and components.

### Client, `@kubb/plugin-client`

One async function per operation, named after the `operationId`, with a typed params object and a
per-call config:

```ts
import { findPetsByTags } from '../gen/clients'

const pets = await findPetsByTags({ tags: ['dog'] }, { signal })
```

By default the function returns the response body. Set the plugin's `client.dataReturnType` to
`'full'` to return the whole response instead. The runtime client lives in a generated
`.kubb/client` module and defaults to the `axios` or `fetch` preset. Point `client.importPath` at
your own module to add a base URL, auth or retries in one place. Confirm these option names in the
plugin's `extension.yaml`.

### React Query, `@kubb/plugin-react-query`

Queries generate `useFindPetsByTags(params, options?)` alongside `findPetsByTagsQueryOptions(params)`
and `findPetsByTagsQueryKey(params)`. Mutations generate `useDeletePet(options?)` with
`deletePetMutationKey()`. The hooks call the generated client, so configure the client once and wrap
the app in a `QueryClientProvider`.

```ts
import { useFindPetsByTags } from '../gen/hooks'

const { data, isPending } = useFindPetsByTags({ tags: ['dog'] })
```

### Vue Query, `@kubb/plugin-vue-query`

Same shape as React Query (`useFindPetsByTags`, `findPetsByTagsQueryOptions`,
`findPetsByTagsQueryKey`), built on `@tanstack/vue-query`. Params can be refs.

### SWR, `@kubb/plugin-swr`

Generates `useFindPetsByTags`, `findPetsByTagsQueryKey` and `findPetsByTagsQueryOptions` on top of
`useSWR`.

### Zod schemas, `@kubb/plugin-zod`

A schema per model (`petSchema`) and per operation (request, params and response schemas). Validate
at runtime and derive a type when you need one:

```ts
import { petSchema } from '../gen/schemas'

const pet = petSchema.parse(json)
type Pet = z.infer<typeof petSchema>
```

Point the client at the schemas to validate responses as they arrive.

### Faker mocks, `@kubb/plugin-faker`

A `create<Model>` factory per model. `createPet()` returns a mock `Pet`; pass overrides to pin
fields. Use it for tests and stories.

### MSW handlers, `@kubb/plugin-msw`

A handler factory per operation, named after the `operationId` (`listPets`). Register the handlers
with `setupServer` in tests or `setupWorker` in the browser, and pass mock data or a resolver.

## Rules

- Never edit files under `output.path`. Change the spec or config and run `kubb generate` again.
- Do not copy generated code into hand-written modules. Import it so a regenerate keeps callers in
  sync.
- Keep the runtime client configured in one place. The client functions and the framework hooks all
  share it.
- After regenerating, run a typecheck so spec changes that broke a call site show up.
